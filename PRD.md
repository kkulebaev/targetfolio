# Targetfolio — Product Requirements Document

**Версия:** 1.0 (MVP, frontend-only)
**Дата:** 2026-04-27
**Статус:** утверждено к реализации

---

## 1. Обзор

Targetfolio — веб-приложение для российских частных инвесторов, использующих Тинькофф Инвестиции. Приложение помогает приближать текущий портфель к целевому распределению весов через рекомендации «что докупить» при заданной сумме свободного кэша.

Текущий PRD описывает **MVP фронтенда без бэкенда**. Реальная интеграция с Tinkoff Invest API — отдельная фаза, в скоуп этого документа не входит.

## 2. Контекст и проблема

Российский частный инвестор регулярно получает кэш для пополнения портфеля (зарплата, дивиденды, продажа активов) и хочет распределять его так, чтобы относительные веса бумаг приближались к личной целевой стратегии. В Тинькове такого инструмента нет — пользователь считает руками или в Excel, теряя время и допуская арифметические ошибки.

## 3. Цели и не-цели

### Цели MVP

- Дать пользователю работающий интерфейс для трёх задач: видеть текущий портфель, задать целевые веса, получить рекомендации по докупке.
- Поддерживать полный рабочий процесс оффлайн, без бэкенда (демо-фикстура + ручной ввод).
- Сохранять целевой портфель и пользовательский реестр инструментов между сессиями.

### Не-цели (вне скоупа MVP)

- Реальная интеграция с Tinkoff Invest API.
- Продажа перекошенных позиций (только докупка).
- Облигации, фьючерсы, валютные позиции в качестве целевых.
- Мульти-валютные портфели и FX-конверсии.
- Несколько именованных целевых портфелей.
- История ребалансов, undo, экспорт.
- Аутентификация, мульти-юзер.
- Налоги, комиссии, ИИС-логика.

## 4. Целевая аудитория

- Активный частный инвестор Тинькофф Инвестиций, работающий с RUB-портфелем (акции МосБиржи + БПИФы).
- Регулярно пополняет портфель (1–4 раза в месяц).
- Имеет осознанную целевую стратегию по весам бумаг.
- Технически грамотен, готов вводить данные руками или работать с демо.

## 5. Ключевые сценарии использования

### 5.1. Первое открытие — демо

Пользователь открывает приложение → видит загруженный демо-портфель (фикстура из 7 позиций) → видит пустую секцию таргета и подсказку «Добавьте позиции в целевой портфель».

### 5.2. Задать целевой портфель

Пользователь добавляет тикеры в секцию «Целевой портфель», задаёт веса. Если бумаги нет в текущем портфеле — добавляет инструмент через диалог «Новый инструмент». При сумме весов = 100% индикатор зелёный.

### 5.3. Получить рекомендации

Пользователь вводит сумму свободного кэша → секция «Рекомендации» автоматически показывает список «купить N лотов X на Y ₽» + неиспользованный остаток.

### 5.4. Ручной ввод портфеля

Пользователь переключает источник в «Ручной ввод» → редактирует таблицу позиций (ticker, name, lotSize, price, quantity) → таргет и рекомендации работают на его данных.

### 5.5. Возврат через сутки

Пользователь закрывает вкладку, открывает заново → `targetWeights` и пользовательский реестр инструментов восстановлены из localStorage; источник сбрасывается в `mock`, текущий портфель — снова демо-фикстура.

## 6. Функциональные требования

### 6.1. Источники данных портфеля

- **FR-1.1.** Поддерживаются два источника: `mock` (фикстура) и `manual` (пользовательский ввод).
- **FR-1.2.** Источник переключается сегментированным контролом в секции «Текущий портфель».
- **FR-1.3.** Данные обоих источников сохраняются параллельно в течение сессии (mock — из JSON, manual — в Pinia).
- **FR-1.4.** На старте приложения активен источник `mock`.

### 6.2. Текущий портфель

- **FR-2.1.** В режиме `mock` таблица read-only, читается из `src/fixtures/portfolio.json`.
- **FR-2.2.** В режиме `manual` таблица редактируема: добавление, удаление, inline-правка строк.
- **FR-2.3.** Колонки таблицы: тикер, название, лот, цена ₽, количество, стоимость ₽ (computed = `quantity × price`), действия.
- **FR-2.4.** Под таблицей отображается суммарная стоимость портфеля.
- **FR-2.5.** Валидация в режиме `manual`:
  - тикер уникален в пределах таблицы и ≥ 1 символа;
  - `lotSize` — целое ≥ 1;
  - `price` > 0;
  - `quantity` — целое ≥ 0.
- **FR-2.6.** Ошибки валидации отображаются inline под полем + красная рамка; кнопка добавления новой строки disabled пока активная строка невалидна.

### 6.3. Целевой портфель

- **FR-3.1.** Таблица: тикер (выбор из реестра инструментов), целевой вес (%), действия.
- **FR-3.2.** Кнопка «+ Добавить тикер» открывает выпадающий список существующих инструментов; при отсутствии нужного — кнопка «Новый инструмент» открывает диалог (ticker, name, lotSize, price), который добавляет инструмент в реестр и автоматически создаёт строку с весом 0.
- **FR-3.3.** Под таблицей всегда отображается сумма весов с цветовым индикатором: зелёный = ровно 100%, янтарный = в диапазоне 95–105 (но не 100), красный = вне диапазона.
- **FR-3.4.** `targetWeights` и пользовательский реестр инструментов персистятся в localStorage.

### 6.4. Рекомендации

- **FR-4.1.** Поле ввода «Свободно для покупки, ₽» сверху секции.
- **FR-4.2.** Расчёт реактивный — пересчитывается при любом изменении входных данных, без явной кнопки.
- **FR-4.3.** Алгоритм: только докупка (без продаж), greedy по рублёвому дефициту относительно таргета, целочисленные лоты с учётом `lotSize`.
- **FR-4.4.** Таблица результата: тикер, название, лотов к покупке, акций к покупке, ориентировочная стоимость ₽.
- **FR-4.5.** Под таблицей отображается `unusedCash` — неиспользованный остаток.
- **FR-4.6.** При невалидном входе таблица скрывается, показывается явное сообщение:
  - сумма весов ≠ 100 → «Сумма целевых весов = X%. Исправьте, чтобы увидеть рекомендации»;
  - пустой таргет → «Добавьте позиции в целевой портфель»;
  - `cashAvailable ≤ 0` → «Введите сумму свободного кэша»;
  - валидно, но `recommendations` пуст → «По текущим параметрам докупать нечего: всё в рамках таргета или денег не хватает на минимальный лот» + отображение `unusedCash`.

### 6.5. Persistence

- **FR-5.1.** Один ключ `localStorage`: `targetfolio:v1` с payload вида `{ _schemaVersion: 1, targetWeights: [...], instruments: [...] }`.
- **FR-5.2.** На загрузке невалидного или несовместимого payload — сброс к дефолтам, `console.warn` с описанием причины.
- **FR-5.3.** **Не персистятся:** `positions`, `cashAvailable`, `source`.
- **FR-5.4.** Реализация — через `pinia-plugin-persistedstate`.

### 6.6. Интерфейс

- **FR-6.1.** Поддержка светлой и тёмной темы с явным переключателем в тулбаре приложения.
- **FR-6.2.** Иконки — `lucide-vue-next`.
- **FR-6.3.** UI на shadcn-vue, style `new-york`, base color `zinc`, Tailwind CSS v4.
- **FR-6.4.** Vue Router не используется — приложение одностраничное.

## 7. Не-функциональные требования

- **NFR-1.** Расчёт рекомендаций синхронный; время отклика на 50 позициях < 50 мс.
- **NFR-2.** TypeScript `strict: true`. В доменном слое (`src/domain/**`) запрещён `any`.
- **NFR-3.** Доменный слой не зависит от Vue, Pinia, DOM, браузерных API. Импорты из `vue` и `pinia` в `src/domain/**` запрещены (контролируется конфигом oxlint, при невозможности — README + ревью).
- **NFR-4.** Покрытие тестами: все ветви `calculateBuys` и функций из `validation.ts` тестируются явно. Coverage-отчёт включён, но без обязательного gate.
- **NFR-5.** Сборка через Vite, лимит размера бандла в MVP не зафиксирован.

### 9.2. Доменная модель (`src/domain/types.ts`)

```ts
export type Ticker = string;

export type Instrument = {
  ticker: Ticker;
  name: string;
  lotSize: number; // акций в лоте; для большинства ETF = 1
  price: number; // RUB, текущая цена за акцию
};

export type Position = {
  ticker: Ticker;
  quantity: number; // в штуках, не лотах
};

export type TargetWeight = {
  ticker: Ticker;
  weightPercent: number; // 0..100
};

export type BuyRecommendation = {
  ticker: Ticker;
  lotsToBuy: number;
  sharesToBuy: number;
  estimatedCost: number; // RUB
};

export type RebalanceResult = {
  recommendations: BuyRecommendation[];
  unusedCash: number;
};
```

### 9.3. Алгоритм `calculateBuys`

**Сигнатура:**

```ts
calculateBuys(
  instruments: Map<Ticker, Instrument>,
  positions: Position[],
  targets: TargetWeight[],
  cash: number,
): RebalanceResult
```

**Шаги:**

1. Построить `currentValueByTicker: Map<Ticker, number>` для всех тикеров из `targets` (отсутствующая позиция = 0).
2. `totalValue = sum(currentValues) + cash`.
3. Для каждого target-тикера: `targetValue = totalValue * weightPercent / 100`; `deficit = targetValue − currentValue`.
4. Цикл: выбрать тикер с максимальным положительным `deficit`. Если `lotCost = lotSize × price ≤ remainingCash` — купить 1 лот, обновить `currentValue` и `deficit` для этого тикера, уменьшить `remainingCash`. Повторять пока ни один лот не помещается в `remainingCash` или все дефициты ≤ 0.
5. Агрегировать покупки по тикеру, сформировать `BuyRecommendation[]` и вернуть `unusedCash = remainingCash`.

**Инварианты:**

- `lotsToBuy` целое > 0 для каждой записи в `recommendations`.
- `sharesToBuy = lotsToBuy × instrument.lotSize`.
- `Σ estimatedCost + unusedCash ≤ cash`.

### 9.4. Pinia сторы

**`usePortfolioStore`** (`src/stores/portfolio.ts`):

- state: `source: 'mock' | 'manual'`, `instruments: Map<Ticker, Instrument>`, `positions: Position[]`.
- actions: `loadFromMock()`, `setSource(s)`, `addInstrument(i)`, `updateInstrument(ticker, patch)`, `removeInstrument(ticker)`, `setManualPositions(p)`, `addPosition(p)`, `updatePosition(ticker, patch)`, `removePosition(ticker)`.
- persisted: `instruments` (только пользовательские; mock-инструменты не персистим).

**`useTargetStore`** (`src/stores/target.ts`):

- state: `targetWeights: TargetWeight[]`.
- actions: `addTicker(ticker)`, `removeTicker(ticker)`, `setWeight(ticker, percent)`.
- getters: `weightsTotal`, `isValid` (`weightsTotal === 100`).
- persisted: `targetWeights`.

**`useRebalanceStore`** (`src/stores/rebalance.ts`):

- state: `cashAvailable: number`.
- getter: `recommendations: RebalanceResult` — вызывает `calculateBuys` из `src/domain/rebalance.ts`, читая первые два стора.
- персистенс: нет.

## 11. Тестирование

### 11.1. Domain-слой (плотное покрытие)

Параметризованные тесты `calculateBuys`:

- пустой портфель + пустой таргет;
- один тикер в таргете 100%;
- портфель ровно совпадает с таргетом (нулевые рекомендации);
- кэш меньше цены любого лота (всё в `unusedCash`);
- сумма весов ≠ 100 (контракт: вход уже валидный, передаём из стора только при `isValid`);
- целочисленность лотов и lotSize > 1;
- тикер в таргете, отсутствующий в позициях;
- перекошенные позиции — greedy выбирает максимальный дефицит первым.

Тесты `validation.ts` — все ветви валидаторов и парсера persisted state.

### 11.2. Pinia-сторы (поверхностно)

По 2–3 теста на каждый стор: загрузка mock, добавление тикера в таргет, валидация суммы весов, корректное прокидывание данных в `recommendations`.

### 11.3. Компоненты

В MVP не покрываем. Точечные тесты на компоненты добавляются по мере появления багов в UI.

### 11.4. Coverage

`vitest run --coverage` — отчёт включён, обязательного gate нет.

## 12. Acceptance criteria

### Алгоритм

- **AC-1.** Функция `calculateBuys` при пустом таргете возвращает `{ recommendations: [], unusedCash: cash }`.
- **AC-2.** Все рекомендации содержат целые `lotsToBuy > 0`; `sharesToBuy = lotsToBuy × lotSize`.
- **AC-3.** `Σ estimatedCost + unusedCash ≤ cash` (никогда не превышает доступный кэш).
- **AC-4.** Тикер из target, отсутствующий в `positions`, обрабатывается как позиция с `quantity = 0`.
- **AC-5.** При совпадении портфеля с таргетом и достаточном кэше алгоритм продолжает докупать, поддерживая пропорции.

### UI

- **AC-6.** Демо-режим работает на первом открытии без действий пользователя.
- **AC-7.** Переключение источника `mock ↔ manual` не теряет данные другого режима в рамках сессии.
- **AC-8.** Перезагрузка страницы восстанавливает `targetWeights` и пользовательский реестр инструментов.
- **AC-9.** Невалидная сумма весов блокирует таблицу рекомендаций с конкретным сообщением, содержащим текущее значение суммы.
- **AC-10.** Светлая и тёмная темы переключаются без визуальных артефактов.
- **AC-11.** В режиме `manual` нельзя добавить две строки с одинаковым тикером.
- **AC-12.** Целевой вес принимает только числовые значения в диапазоне 0–100.

## 13. Out of scope / Roadmap

После закрытия MVP в порядке убывания приоритета:

1. Бэкенд + интеграция с Tinkoff Invest API (загрузка реального портфеля по read-only токену).
2. Несколько именованных целевых портфелей с CRUD.
3. Полный ребаланс с продажами (с учётом налогов и комиссий).
4. Облигации (НКД, котировка в % от номинала, дюрация).
5. Мульти-валютный портфель + источник FX-курсов.
6. История ребалансов и undo.
7. Экспорт рекомендаций (CSV / печатная версия).
8. E2E-тесты (Playwright).
9. CI/CD.
